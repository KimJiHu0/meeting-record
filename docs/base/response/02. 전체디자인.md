# MeetingFlow 전체 디자인 기획서

작성일: 2026-06-18  
문서 목적: 다음 퍼블리셔가 MeetingFlow의 전역 CSS, 디자인 토큰, 공통 컴포넌트 스타일을 일관되게 설계할 수 있도록 전체 디자인 기준을 정의한다.

## 1. 디자인 목표

디자인의 핵심 목표는 사용자가 별도의 사용 가이드를 읽지 않아도 다음 행동을 바로 이해하게 만드는 것이다. 화면은 화려한 장식보다 명확한 흐름, 읽기 쉬운 정보 구조, 신뢰감 있는 상태 표현을 우선한다.

### 1.1 핵심 경험

-   회의 등록, 녹음, 업로드, 분석, 일정 저장 흐름이 끊기지 않아야 한다.
-   사용자는 현재 상태가 무엇인지, 다음에 무엇을 해야 하는지 즉시 알 수 있어야 한다.
-   개발자 대상 서비스지만 복잡한 업무툴처럼 무겁게 보이지 않아야 한다.
-   개인 프로젝트에서 매일 쓰는 도구처럼 가볍고 빠르게 느껴져야 한다.

### 1.2 디자인 키워드

-   간결함: 기능은 많아도 화면의 첫인상은 단순해야 한다.
-   명확함: 버튼, 상태, 필터, 결과 영역의 역할이 한눈에 구분되어야 한다.
-   신뢰감: AI 분석, 일정 생성, 저장 상태가 안정적으로 보이도록 색과 문구를 절제한다.
-   생기: 칙칙한 회색 업무툴이 아니라 파릇하고 산뜻한 생산성 도구처럼 보이게 한다.
-   실행 중심: 보기 좋은 대시보드보다 사용자가 바로 회의를 시작하고 결과를 처리하는 흐름을 우선한다.

## 2. 레퍼런스 해석: 토스

디자인 레퍼런스는 토스로 한다. 단, 토스 UI를 그대로 모방하지 않고 MeetingFlow의 도메인에 맞게 다음 원칙을 가져온다.

### 2.1 가져올 점

-   넓은 여백과 높은 가독성
-   강한 대비의 핵심 CTA
-   짧고 직접적인 문구
-   복잡한 정보를 단계별로 쪼개 보여주는 구조
-   사용자가 망설이지 않도록 다음 행동을 분명히 제시하는 인터페이스
-   회색 계열을 많이 쓰더라도 화면이 탁해 보이지 않는 밝은 배경 운용

### 2.2 피할 점

-   금융 앱처럼 지나치게 소비자 앱 톤으로 보이는 것
-   단순함을 이유로 정보 밀도를 너무 낮추는 것
-   모든 영역을 둥근 카드로 감싸 화면이 조각나 보이는 것
-   블루만 강하게 사용해 흔한 SaaS 대시보드처럼 보이는 것

## 3. 톤앤매너

전체 톤은 "파릇한 신뢰감"으로 정의한다. 흰색과 밝은 회색을 기반으로 하되, 메인 시그니처 컬러는 생기 있는 그린 계열을 사용한다. 그린은 회의 후 실행, 완료, 정리의 이미지를 만들고, 생산성 서비스의 가벼운 느낌을 강화한다.

### 3.1 시그니처 컬러

-   이름: Flow Green
-   HEX: `#10B981`
-   용도: 주요 CTA, 활성 탭, 포커스 링, 핵심 상태 강조, 선택된 필터
-   인상: 산뜻함, 진행감, 완료감, 안정감

### 3.3 색상 사용 비율

-   배경/표면: 70%
-   텍스트/경계선: 20%
-   메인 컬러와 상태 컬러: 10%

메인 컬러는 많이 칠하기보다 중요한 행동과 상태를 정확히 표시하는 데 사용한다. 넓은 초록 면적을 남발하면 서비스가 가벼워 보일 수 있으므로 CTA, 활성 상태, 완료 상태 중심으로 제한한다.

## 4. 타이포그래피

### 4.1 폰트 방향

기본 폰트는 시스템 sans-serif를 사용한다. Next.js 기본 템플릿의 Geist 계열을 사용할 수 있다면 우선 사용하고, 별도 폰트 도입 없이도 충분히 정돈된 인상을 만든다.

```css
font-family: var(--font-geist-sans), -apple-system, BlinkMacSystemFont, 'Segoe UI', sans-serif;
```

### 4.2 타입 스케일

| 역할          | 크기 | 행간 | 굵기 | 사용처           |
| ------------- | ---: | ---: | ---: | ---------------- |
| Page title    | 28px | 36px |  700 | 화면 최상단 제목 |
| Section title | 22px | 30px |  700 | 주요 섹션 제목   |
| Card title    | 18px | 26px |  700 | 카드/패널 제목   |
| Body          | 15px | 24px |  400 | 기본 본문        |
| Body strong   | 15px | 24px |  600 | 강조 본문        |
| Caption       | 13px | 20px |  400 | 메타, 설명       |
| Label         | 13px | 18px |  600 | 입력 라벨, 배지  |
| Button        | 15px | 20px |  700 | 주요 버튼        |

### 4.3 문구 원칙

-   버튼 문구는 동사 중심으로 작성한다. 예: `회의 시작`, `분석 요청`, `일정 저장`
-   상태 문구는 현재 상태를 먼저 말한다. 예: `분석 중`, `저장 완료`, `업로드 실패`
-   설명 문구는 한 줄을 우선한다. 길어지면 툴팁이나 상세 영역으로 분리한다.

## 5. 레이아웃 원칙

### 5.1 전체 구조

앱은 데스크톱 우선으로 설계하되 Phone, Tablet, Desktop 반응형을 반드시 고려한다.

-   Desktop: 사이드 내비게이션 + 메인 콘텐츠 구조를 기본으로 한다.
-   Tablet: 사이드 내비게이션은 축소하거나 상단 내비게이션으로 전환한다.
-   Phone: 하단 또는 상단 중심의 단일 컬럼 구조로 전환한다.

### 5.2 Breakpoint

```css
--breakpoint-phone: 0px;
--breakpoint-tablet: 768px;
--breakpoint-desktop: 1024px;
--breakpoint-wide: 1280px;
```

| 구간    |        기준 | 레이아웃                            |
| ------- | ----------: | ----------------------------------- |
| Phone   |     0-767px | 단일 컬럼, 주요 CTA 하단 고정 가능  |
| Tablet  |  768-1023px | 단일 컬럼 또는 2컬럼, 사이드바 축소 |
| Desktop | 1024px 이상 | 사이드바 + 콘텐츠, 12컬럼 기반      |
| Wide    | 1280px 이상 | 최대 콘텐츠 폭 제한                 |

### 5.3 콘텐츠 폭

-   앱 메인 콘텐츠 최대 폭: `1200px`
-   상세 화면 읽기 영역 최대 폭: `840px`
-   채팅 화면 본문 최대 폭: `920px`
-   폼 화면 최대 폭: `480px`

넓은 화면에서 콘텐츠를 무작정 늘리지 않는다. 회의 상세의 STT, 요약, Action Item처럼 읽기 중심 화면은 폭을 제한해 시선을 안정시킨다.

### 5.4 간격 시스템

4px 기반 spacing scale을 사용한다.

```css
--space-1: 4px;
--space-2: 8px;
--space-3: 12px;
--space-4: 16px;
--space-5: 20px;
--space-6: 24px;
--space-8: 32px;
--space-10: 40px;
--space-12: 48px;
--space-16: 64px;
```

기본 규칙:

-   화면 바깥 padding: Phone `20px`, Tablet `28px`, Desktop `32px`
-   섹션 간격: `32px`
-   카드 내부 padding: `20px` 또는 `24px`
-   폼 요소 간격: `12px`
-   버튼 그룹 간격: `8px`

## 6. 형태와 깊이

### 6.1 Radius

둥근 느낌은 부드럽게 사용하되 과하게 귀엽게 만들지 않는다.

```css
--radius-xs: 6px;
--radius-sm: 8px;
--radius-md: 12px;
--radius-lg: 16px;
--radius-full: 999px;
```

사용 기준:

-   버튼, 입력창: `8px`
-   카드, 패널: `12px`
-   모달, 큰 업로드 영역: `16px`
-   배지, pill 탭: `999px`

### 6.2 Shadow

기본 화면은 shadow보다 경계선과 배경 차이로 구조를 만든다. shadow는 떠 있는 UI에만 제한적으로 사용한다.

```css
--shadow-sm: 0 1px 2px rgba(17, 24, 39, 0.06);
--shadow-md: 0 8px 24px rgba(17, 24, 39, 0.08);
--shadow-lg: 0 16px 40px rgba(17, 24, 39, 0.12);
```

사용 기준:

-   일반 카드: shadow 없이 border 사용
-   드롭다운, 모달: `--shadow-md`
-   중요 플로팅 CTA: `--shadow-lg`

## 7. 공통 컴포넌트 스타일

### 7.1 버튼

버튼은 사용자의 다음 행동을 가장 분명하게 만드는 요소다. 한 화면의 Primary Button은 원칙적으로 1개만 둔다.

#### Primary

-   배경: `--color-primary`
-   텍스트: `#FFFFFF`
-   높이: `44px`
-   padding: `0 18px`
-   radius: `8px`
-   hover: `--color-primary-hover`
-   disabled: 배경 `#D1D5DB`, 텍스트 `#FFFFFF`

사용처: 회의 시작, 분석 요청, 일정 저장, 로그인

#### Secondary

-   배경: `--color-surface`
-   텍스트: `--color-text`
-   border: `1px solid --color-border`
-   hover 배경: `--color-surface-muted`

사용처: 오디오 업로드, 취소, 보조 이동

#### Ghost

-   배경: transparent
-   텍스트: `--color-text-muted`
-   hover 배경: `--color-surface-muted`

사용처: 상세 보기, 더보기, 부가 액션

#### Danger

-   배경: `#FEF2F2`
-   텍스트: `--color-danger`
-   hover 배경: `#FEE2E2`

사용처: 삭제, 분석 결과 초기화

### 7.2 카드와 패널

카드는 정보를 담는 최소 단위로 사용한다. 모든 섹션을 카드로 감싸지 않는다.

기본 카드:

-   배경: `--color-surface`
-   border: `1px solid --color-border-soft`
-   radius: `12px`
-   padding: `20px`
-   shadow: 없음

강조 카드:

-   배경: `--color-primary-subtle`
-   border: `1px solid #A7F3D0`
-   radius: `16px`
-   padding: `24px`

사용처:

-   최근 회의 요약
-   분석 결과 요약
-   일정 후보
-   Action Item

카드 내부는 제목, 메타, 내용, 액션 순서로 통일한다.

### 7.3 입력창

입력창은 명확하고 조용해야 한다. 과한 그림자나 두꺼운 테두리는 사용하지 않는다.

-   높이: `44px`
-   padding: `0 14px`
-   radius: `8px`
-   배경: `--color-surface`
-   border: `1px solid --color-border`
-   placeholder: `--color-text-subtle`
-   focus border: `--color-primary`
-   focus ring: `0 0 0 3px rgba(16, 185, 129, 0.14)`

텍스트 영역:

-   min-height: `120px`
-   padding: `14px`
-   line-height: `24px`

### 7.4 탭

탭은 회의 상세의 `요약`, `결정사항`, `Action Item`, `Transcript` 전환에 사용한다.

Pill 탭:

-   높이: `36px`
-   padding: `0 14px`
-   radius: `999px`
-   inactive 텍스트: `--color-text-muted`
-   active 배경: `--color-primary-soft`
-   active 텍스트: `#047857`

Underline 탭:

-   상세 페이지의 큰 콘텐츠 전환에 사용한다.
-   active border-bottom: `2px solid --color-primary`
-   active 텍스트: `--color-text`

### 7.5 배지

상태 배지는 짧은 단어와 색으로 현재 상태를 알려준다.

| 상태      | 텍스트    |      배경 | 텍스트 색 |
| --------- | --------- | --------: | --------: |
| 녹음 전   | 준비됨    | `#F3F4F6` | `#4B5563` |
| 녹음 중   | 녹음 중   | `#FEF2F2` | `#DC2626` |
| 업로드 중 | 업로드 중 | `#EFF6FF` | `#2563EB` |
| 분석 중   | 분석 중   | `#EFF6FF` | `#2563EB` |
| 분석 완료 | 완료      | `#D1FAE5` | `#047857` |
| 실패      | 실패      | `#FEF2F2` | `#DC2626` |
| 저장됨    | 저장됨    | `#ECFDF5` | `#059669` |

### 7.6 모달

-   Desktop: 중앙 모달, max-width `560px`
-   Phone: 하단 시트 형태 권장
-   배경 dim: `rgba(17, 24, 39, 0.42)`
-   모달 radius: `16px`
-   padding: `24px`

삭제 확인, 일정 상세, 회의 수정처럼 현재 흐름을 잠시 멈춰야 하는 작업에만 사용한다.

### 7.7 파일 업로드 영역

오디오 업로드는 사용자가 불안해하기 쉬운 기능이므로 영역과 상태를 명확히 보여준다.

-   배경: `--color-surface`
-   border: `1px dashed #A7F3D0`
-   hover 배경: `--color-primary-subtle`
-   radius: `16px`
-   padding: `32px`
-   아이콘/문구/보조 설명/파일 선택 버튼 순서

업로드 중에는 진행률 바와 파일명을 함께 표시한다.

### 7.8 오디오 플레이어와 녹음 UI

녹음 UI는 화면에서 가장 강한 시각적 중심이 되어야 한다.

-   타이머: 32px 이상, 굵기 700
-   녹음 상태 점: red 계열, 1초 내외의 부드러운 pulse 가능
-   녹음 시작 버튼: Primary 또는 큰 원형 버튼
-   종료 버튼: Danger 스타일
-   일시정지 기능이 있다면 Secondary 스타일

Phone에서는 녹음 버튼을 화면 하단에 고정할 수 있다.

## 8. 화면별 디자인 기준

### 8.1 로그인/회원가입

-   폼 폭은 `420px` 내외로 제한한다.
-   브랜드명과 한 줄 설명을 상단에 둔다.
-   입력 필드, 에러 메시지, CTA 순서로 단순하게 구성한다.
-   회원가입과 로그인 전환은 보조 링크로 처리한다.

### 8.2 대시보드

대시보드는 "오늘 무엇을 처리해야 하는지"와 "새 회의를 어떻게 시작하는지"가 즉시 보여야 한다.

우선순위:

1. 새 회의 생성 또는 녹음 시작 CTA
2. 최근 회의와 분석 상태
3. 예정 일정
4. 빠른 업로드

추천 구조:

-   상단: 인사말, 현재 요약, Primary CTA
-   중단: 최근 회의 목록
-   우측 또는 하단: 예정 일정, 분석 대기 항목

### 8.3 회의 목록

-   검색과 상태 필터를 상단에 배치한다.
-   목록은 카드보다 테이블형 리스트에 가깝게 구성해 정보 밀도를 확보한다.
-   각 행은 제목, 날짜, 상태, 주요 액션만 보여준다.
-   상세 이동은 행 전체 클릭 또는 명확한 버튼으로 제공한다.

### 8.4 회의 생성/녹음

-   화면의 중심은 녹음 상태와 타이머다.
-   회의 제목 입력은 녹음 전 필수 정보로 자연스럽게 배치한다.
-   녹음 시작, 종료, 저장 순서가 시각적으로 분명해야 한다.
-   녹음 중에는 다른 부가 기능보다 현재 녹음 상태를 우선한다.

### 8.5 오디오 업로드

-   파일 선택 영역을 크게 보여준다.
-   업로드 가능 형식, 최대 용량, 예상 소요 시간을 보조 문구로 제공한다.
-   업로드 완료 후 다음 CTA는 `분석 요청`으로 바뀐다.

### 8.6 회의 상세/AI 분석 결과

회의 상세는 가장 복잡한 화면이므로 정보 위계를 엄격히 지킨다.

상단:

-   회의 제목
-   날짜/시간
-   분석 상태 배지
-   주요 액션: 분석 요청, 일정 저장, 수정

본문:

-   요약은 가장 먼저 노출한다.
-   결정사항과 Action Item은 리스트 형태로 분리한다.
-   일정 후보는 저장 액션이 가장 잘 보이도록 구성한다.
-   Transcript는 긴 텍스트이므로 접기, 검색, 탭 분리를 고려한다.

### 8.7 일정 목록

-   날짜 필터와 일정 목록을 명확히 분리한다.
-   일정은 시간, 제목, 출처 회의, 상태 순서로 보여준다.
-   AI가 생성한 일정은 출처를 표시해 신뢰감을 준다.

### 8.8 AI 질문

-   채팅 인터페이스는 단순하게 유지한다.
-   회의 선택 필터를 상단에 둔다.
-   입력창은 하단 고정 구조를 권장한다.
-   AI 답변에는 참조한 회의나 일정이 있다면 작은 출처 링크를 제공한다.

## 9. 상태 표현 원칙

서비스의 핵심은 비동기 처리다. 녹음, 업로드, AI 분석, 일정 저장 상태가 명확해야 한다.

### 9.1 Loading

-   버튼 내부 spinner와 문구를 함께 사용한다. 예: `분석 중...`
-   전체 화면 skeleton보다 해당 영역 skeleton을 우선한다.
-   2초 이상 걸릴 수 있는 작업은 진행 상태 문구를 제공한다.

### 9.2 Empty

빈 상태는 다음 행동을 함께 제공한다.

예시:

-   회의가 없음: `아직 등록된 회의가 없습니다.` + `새 회의 만들기`
-   일정이 없음: `저장된 일정이 없습니다.` + `회의 분석 결과에서 일정을 저장해보세요.`
-   질문 내역 없음: `궁금한 내용을 질문해보세요.` + 입력창 강조

### 9.3 Error

에러는 원인과 복구 행동을 함께 보여준다.

예시:

-   `오디오 업로드에 실패했습니다. 파일 형식을 확인한 뒤 다시 시도해주세요.`
-   `AI 분석을 완료하지 못했습니다. 잠시 후 다시 요청해주세요.`

에러 색은 위험 작업에만 강하게 사용한다. 일반 안내성 오류는 문구와 아이콘으로 충분히 전달한다.

### 9.4 Success

성공 상태는 짧고 조용하게 표시한다.

예시:

-   `일정이 저장되었습니다.`
-   `분석이 완료되었습니다.`

toast 또는 inline success message를 사용하고, 화면 전체를 가리지 않는다.

## 10. 접근성과 사용성 기준

Next.js는 라우트 이동 시 접근성 보조 기능을 제공하므로 각 페이지는 고유하고 명확한 제목을 가져야 한다. 화면마다 `h1`은 하나만 두고, 라우트별 title이 의미 있게 설정되도록 퍼블리싱한다.

기본 기준:

-   텍스트와 배경의 대비는 WCAG AA 수준을 목표로 한다.
-   모든 버튼은 키보드 포커스 상태가 보여야 한다.
-   아이콘 단독 버튼은 `aria-label` 또는 시각적으로 숨긴 텍스트를 제공한다.
-   색상만으로 상태를 전달하지 않는다. 텍스트 배지를 함께 사용한다.
-   애니메이션은 짧게 사용하고 `prefers-reduced-motion`을 고려한다.
-   클릭 가능한 영역은 최소 `40px` 이상의 높이를 확보한다.

## 11. 반응형 기준

### 11.1 Phone

-   단일 컬럼
-   화면 좌우 padding `20px`
-   목록은 카드형 또는 간단한 row 형태
-   주요 CTA는 하단 고정 가능
-   테이블은 카드 리스트로 변환

### 11.2 Tablet

-   1컬럼과 2컬럼을 화면 성격에 따라 혼합
-   사이드바는 축소형 또는 상단 내비게이션으로 전환
-   상세 화면은 요약과 액션 영역을 먼저 보여준다.

### 11.3 Desktop

-   사이드바 + 메인 콘텐츠
-   대시보드는 2컬럼 또는 3컬럼 구성 가능
-   회의 상세는 본문과 보조 패널을 분리 가능
-   콘텐츠 최대 폭을 제한해 과도한 가로 확장을 막는다.

## 15. 최종 디자인 방향 요약

MeetingFlow의 디자인은 토스처럼 쉽고 명확하지만, 더 개발자 생산성 도구에 맞는 차분한 정보 구조를 가져야 한다. 전체 인상은 흰색 기반의 깨끗한 표면, 파릇한 Flow Green 포인트, 부드러운 경계선, 짧고 직접적인 문구로 완성한다.

퍼블리셔는 이 문서를 기준으로 먼저 전역 CSS 변수와 공통 컴포넌트 스타일을 잡고, 이후 각 화면을 `사용자의 다음 행동이 가장 잘 보이는가`라는 기준으로 구현한다.
